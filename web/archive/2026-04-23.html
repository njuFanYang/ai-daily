    Codex 用 WebSocket 砍掉 Agent 循环延迟

    openai_blog agent_architecturellm_engineering

    揭秘 Codex 连接级缓存与 WebSocket 优化，干货工程细节

    OpenAI 在 Responses API 中引入 WebSocket 传输，替代原有的 HTTP 长连接，用于 Codex agent 循环中的多轮交互。通过连接级缓存（connection-scoped caching），同一 session 内的 prompt prefix、tool 定义和 context 无需在每轮重复序列化与校验，显著削减了 API 调用的协议开销和首 token 延迟。对于需要频繁调用 tool use、迭代执行的 agent 工作流，这种改造能让单轮之间的"黏连"更紧，端到端响应更接近本地推理的节奏。原文披露了 Codex 内部 agent loop 的落地细节，包括连接生命周期管理、缓存失效策略如何与 prompt caching 协同。对于正在自研 agent 框架、受制于 HTTP 轮询延迟或 token 重传成本的工程师，值得参考其传输层优化思路；若只使用单轮 chat completion，增量价值有限。具体的延迟降幅数字、吞吐对比和 SDK 接入方式原文未在摘要中透露，需点进原文查看基准测试数据。

    HELM：给 VLA 模型补上长程记忆与自检

    arxiv_cs_lg agent_architecturemultimodal

    直指 VLA 长程任务三大缺陷，给出记忆+验证+恢复新框架

    VLA 模型在长程操控任务上系统性失败，根源不是 context window 不够长，而是 reactive 执行回路里三个缺口：memory gap、verification gap、recovery gap。HELM 是一个与底层模型解耦的框架，用三件套补齐：Episodic Memory Module（EMM）通过 CLIP 索引的关键帧检索任务历史；State Verifier（SV）在动作执行前，基于观测、动作、子目标和 memory 条件上下文预测失败；Harness Controller（HC）负责回滚和 replanning。核心学习贡献是 SV，作者声称它稳定优于基于规则的可行性检查和 ensemble 不确定性 baseline，而且离开 episodic memory 就明显掉点，说明"记忆"不是装饰而是 verifier 能工作的前提。原文摘要在 LIBERO 基准处被截断，具体数字未透露，但工程上的启示很清楚：给 VLA 套一层可回滚的 harness，比单纯堆长上下文或训更大模型更能解决长程失败，适合正在做机器人或 agent 执行可靠性的工程师一读。
